What is a SAGA?
// formal definition, motivation, anatomy
The Problem
A business transaction that spans multiple microservices — each with its own database — cannot use a single ACID transaction.
The Solution
Break the global transaction into a sequence of local transactions T₁…Tₙ, each committing immediately, with compensating transactions C₁…Cₙ₋₁ for undo semantics.
The Invariant
Either all Tᵢ succeed (happy path) or Cᵢ…C₁ execute in reverse order (compensating path). Partial completion is never the final state.
Happy path — 4 services, 4 local transactions
Failure path — T3 fails, compensations unwind backwards
Formal definition (Garcia-Molina & Salem, 1987)

A SAGA is a sequence of transactions T₁, T₂, …, Tₙ that can be interleaved with other transactions. The database system guarantees that either all transactions in the SAGA are executed, or if the SAGA does not complete, then the sub-sequence T₁, T₂, …, Tₖ, Cₖ, Cₖ₋₁, …, C₁ is executed for some 0 ≤ k < n. Each Cᵢ is the compensating transaction for Tᵢ.

Choreography-based SAGA
// event-driven · no central coordinator · pub/sub
Architecture principle

Each service listens to events from other services and decides locally what to do next. There is no central "brain" — the workflow emerges from event reactions. Services are fully decoupled at the deployment level but implicitly coupled via the event contract.

Orchestration-based SAGA
// central coordinator · command/reply pattern · explicit state machine
Architecture principle

A dedicated Saga Orchestrator issues commands to participant services and reacts to their replies. The orchestrator owns the entire saga state machine. Services have no knowledge of each other — they only respond to commands from the orchestrator. This makes the workflow explicit and auditable but introduces the orchestrator as a potential SPOF.

Compensating Transactions
// semantic undo · idempotency · not a rollback
Why compensations ≠ database rollbacks
Database Rollback
  • Atomic — as-if-never-happened
  • Invisible to other transactions
  • Done by the DB engine
  • No business logic needed
  • Instant, guaranteed
Compensating Transaction
  • A new forward transaction
  • Effects are visible in the window T–C
  • Written by the developer
  • Requires business logic
  • Can itself fail and need retry
Compensation matrix — e-commerce order saga
#TransactionAction Compensating TxIdempotent?Notes
T1 Create Order status=PENDING C1: status=CANCELLED Yes Setting status is idempotent
T2 Reserve Payment debit hold account C2: release hold → credit Yes Use idempotency key on payment gateway
T3 Reserve Inventory qty -= n C3: qty += n Careful Must prevent double-compensation
T4 Create Shipment allocate carrier slot C4: cancel shipment External Carrier API may not be idempotent — add dedup
T5 Send Confirmation email dispatched C5: n/a (send apology) Impossible Cannot un-send email — pivot transaction
Compensation failure strategies
Retry with exponential backoff
Compensations MUST be retried until they succeed. Design every Cᵢ to be idempotent from day one.
Dead-letter queue (DLQ)
Exhausted retries → DLQ → alerting → human or automated replay. Never silently discard.
Durable saga state store
Persist which step the saga is on so any process can resume after a crash — at-least-once execution.
Pivot transaction
Some actions (sent email, printed invoice) cannot be undone. Pivot = last non-compensatable step; design all prior steps to be compensatable.
The Lost Isolation Problem
// dirty reads · lost updates · countermeasures
Why SAGAs have no Isolation

In a classic ACID transaction, the I guarantees that intermediate states are invisible to other transactions. A SAGA has no such guarantee — each Tᵢ commits immediately and becomes visible to the entire world before the saga completes.

Anomalies and countermeasures
AnomalyDescriptionCountermeasure
Dirty read Another saga reads a value written by a still-in-progress saga that later compensates Semantic locks — mark records as "in saga" and refuse reads from other sagas
Lost update Two sagas read the same record, both modify it; one overwrites the other's change Optimistic locking with version numbers / ETags
Fuzzy / non-repeatable read Same query returns different results within one saga because another saga committed between the reads Re-read and verify at critical decision points; use snapshot-at-saga-start
Phantom read A range query returns different rows across two reads within the same saga Mostly accepted as a known SAGA trade-off; mitigate with order-level locking
Semantic lock pattern

A simple countermeasure: add a saga_lock flag / state to records being modified. Any concurrent saga that tries to read a locked record either fails fast or retries. The lock is released when the owning saga completes or compensates.

-- T1: Create order with semantic lock INSERT INTO orders (id, status, saga_lock) VALUES ('ord-123', 'PENDING', TRUE); -- Concurrent reader checks the lock SELECT * FROM orders WHERE id = 'ord-123' AND saga_lock = FALSE; -- Returns 0 rows → caller retries or fails fast -- C1: Release lock on compensation UPDATE orders SET status = 'CANCELLED', saga_lock = FALSE WHERE id = 'ord-123';

Trade-offs & Comparisons
// SAGA vs 2PC vs Outbox · choreography vs orchestration

✦ Pros

  • No distributed lock contention
  • Each service stays loosely coupled
  • Works across heterogeneous DBs
  • Scales independently per service
  • Survives partial network failures
  • Long-running transactions OK

✦ Cons

  • No isolation — dirty reads possible
  • Compensations add business complexity
  • Debugging distributed state is hard
  • Choreography: implicit event coupling
  • Orchestration: orchestrator = SPOF risk
  • Testing all failure paths is expensive

✦ Use when

  • Multiple microservices need coordination
  • Long-lived business transactions
  • Services own separate databases
  • High throughput (2PC is too slow)
  • Cross-cloud / cross-vendor services

✦ Avoid when

  • Strong isolation required (financial ledger)
  • Single service / single DB — use ACID
  • Compensations are impossible (email)
  • Team unfamiliar with eventual consistency
